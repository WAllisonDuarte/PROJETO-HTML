Dicionário de API para Novos Desenvolvedores

Temas Sobre Profissional

Tipos de API

  • REST: Baseada em HTTP e geralmente usa JSON para comunicação.
  • SOAP: Utiliza XML para troca de mensagens e tem regras mais rígidas.
  • GraphQL: Permite consultas mais flexíveis, onde o cliente especifica os dados que deseja receber.
  • WebSockets: Mantém conexões abertas para comunicação bidirecional em tempo real.
  • gRPC: Usa protocolo HTTP/2 e se baseia em contratos definidos com Protocol Buffers.

Endpoints

  • Exemplo de endpoint em uma API REST:
    • GET https://api.exemplo.com/users
  • Base URL: https://api.exemplo.com
  • Recurso: /users
  • Parâmetros (opcionais): /users?id=1

Autenticação e Autorização

  • Autenticação: Garante que um usuário é quem diz ser (ex: JWT, OAuth, API Key).
  • Autorização: Define quais recursos um usuário pode acessar.
  • Métodos comuns:
    • API Key (chave de API)
    • OAuth 2.0 (utilizado por plataformas como Google e Facebook)
    • JWT (JSON Web Token)

Request e Response

  • Request (Requisição): Envio de dados para a API 
    GET /users/1 HTTP/1.1
Host: api.exemplo.com
Authorization: Bearer token
  • Response (Resposta): Retorno da API. 
    {
    "id": 1,
    "name": "João Silva",
    "email": "joao@email.com"
}
  • Códigos de status HTTP comuns:
    • 200 OK - Sucesso
    • 201 Created - Recurso criado
    • 400 Bad Request - Erro na requisição
    • 401 Unauthorized - Falta de autenticação
    • 404 Not Found - Recurso não encontrado

Rate Limiting

  • Controle da quantidade de requisições permitidas por um período.
  • Exemplo de resposta ao exceder o limite: 
    {
    "error": "Too Many Requests",
    "retry_after": 30
}

Versionamento e Documentação

  • Versionamento: Manter diferentes versões da API para compatibilidade.
  • Exemplo de versão no endpoint: https://api.exemplo.com/v1/users
  • Documentação: Explicação clara do uso da API.
    • Ferramentas comuns: Swagger (OpenAPI), Postman, API Blueprint

Webhooks

  • Mecanismo para APIs notificarem eventos de forma assíncrona.
  • Exemplo: Quando um pagamento é aprovado, um webhook pode notificar um sistema externo.
  • {
    "event": "payment_success",
    "amount": 100.00,
    "user": "joao@email.com"
}

Testes de API e Caching

  • Testes de API:
    • Ferramentas: Postman, Jest, Newman.
    • Testes unitários, de integração e de carga.
  • Caching de API:
    • Evita requisições desnecessárias para otimizar performance.
    • Ferramentas: Redis, Varnish, Cloudflare.

API Gateway

  • Intermediário entre cliente e APIs internas.
  • Benefícios:
    • Segurança (autenticação, autorização)
    • Controle de tráfego (rate limiting, caching)
    • Balanceamento de carga

GraphQL

  • Alternativa ao REST que permite ao cliente definir exatamente os dados que precisa.
  • Exemplo de requisição GraphQL: 
    query {
    user(id: 1) {
        name
        email
    }
}
  • Exemplo de resposta: 
    {
    "data": {
        "user": {
            "name": "João Silva",
            "email": "joao@email.com"
        }
    }
}

Pipeline de Dados

O pipeline de dados automatiza a movimentação de dados entre diferentes sistemas, garantindo sua coleta, transformação e armazenamento de forma contínua.

Etapas de um Pipeline:

  • Fonte de Dados: Sistemas de onde os dados são coletados.
  • ETL/ELT: Processo de transformação dos dados.
  • Destino Final: Data Warehouse ou Data Lake.

ETL vs ELT

ETL (Extract, Transform, Load): Os dados são extraídos, transformados e depois carregados no destino.

ELT (Extract, Load, Transform): Os dados são extraídos e carregados diretamente no destino, e a transformação ocorre posteriormente.

Diferença Principal:

  • ETL: Adequado para data warehouses tradicionais.
  • ELT: Melhor para data lakes e ambientes modernos.

Data Warehousing

Um data warehouse é um repositório centralizado que organiza e armazena dados estruturados para consultas rápidas e análise de dados.

Principais características:

  • Dados estruturados.
  • Focado em análise histórica e relatórios.
  • Exemplos de ferramentas: Snowflake, Amazon Redshift, Google BigQuery.

Data Lake

Um data lake armazena dados em seu formato original (estruturado, semiestruturado e não estruturado). É mais flexível, mas requer governança adequada.

Diferença entre Data Lake e Data Warehouse:

  • Data Warehouse: Dados estruturados, organizados para consultas.
  • Data Lake: Dados brutos, sem estrutura definida.

Big Data

Big Data refere-se a conjuntos massivos de dados que requerem tecnologias especiais para armazenamento e processamento.

Os 5 Vs do Big Data:

  • Volume: Grande quantidade de dados.
  • Velocidade: Dados gerados em alta frequência.
  • Variedade: Diferentes formatos de dados.
  • Veracidade: Dados confiáveis e precisos.
  • Valor: Insights obtidos dos dados.

Armazenamento de Dados

O armazenamento de dados envolve escolher a melhor tecnologia para guardar informações de forma segura e acessível.

Principais tipos de armazenamento:

  • Bancos de Dados Relacionais (SQL): Oracle, MySQL, PostgreSQL.
  • Bancos de Dados NoSQL: MongoDB, Cassandra, Redis.
  • Armazenamento em Nuvem: AWS S3, Azure Blob Storage, Google Cloud Storage.

ETL:

  • Informatica

Big Data:

  • Hadoop

  • Kafka

Data Warehouse:

  • Snowflake
  • Amazon Redshift

Data Lake:

  • AWS S3
  • Azure Data Lake
  • Google Cloud Storage

Qualidade de Dados

A qualidade de dados refere-se à precisão, consistência e integridade das informações.

Principais aspectos:

  • Validade: Os dados estão no formato correto?
  • Consistência: Dados coerentes entre sistemas.
  • Integridade: Dados completos e sem lacunas.

Arquitetura de Dados

A arquitetura de dados define como os dados são coletados, armazenados e processados dentro de uma organização.

Principais componentes:

  • Fonte de Dados: Origem das informações.
  • Pipeline de Dados: Fluxo de processamento.
  • Armazenamento: Data lake ou data warehouse.
  • Ferramentas de análise: Power BI, Tableau.

Arquitetura Serverless

Na arquitetura serverless, o desenvolvedor não precisa gerenciar servidores diretamente. A infraestrutura é escalada automaticamente.

Benefícios:

  • Menos custos operacionais.
  • Escalabilidade automática.
  • Foco no código e nas funcionalidades.
